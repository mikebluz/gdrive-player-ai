# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## UI guidance

Apply these rules to every UI change:

1. **No horizontal scrolling** — with ONE intentional exception (below). If a row's contents don't fit, either (a) shrink elements in that row to fit (`min-width: 0`, `flex-shrink: 1`, narrower padding/font on small viewports via media queries), or (b) wrap to a new row and resize the new row's contents to fit. Never solve overflow with `overflow-x: auto/scroll`.
   - **Exception — lane step strips (`.lane-chips`).** Each lane's step strip is a single-row, proportional-width **timeline** that intentionally scrolls horizontally (`overflow-x: auto`) and fills logically to the right; during playback each lane auto-scrolls to keep its currently-playing step centered. This is deliberate (chosen over wrapping) so there are no row-boundary / continuation-split artifacts. Do NOT "fix" it back into a wrapping layout.

2. **Audio playback must be smooth and immediate.** Every user-triggered sound (cell press, wrap audition, sequence/loop playback, REST tap, sample preview, recording playback, etc.) should fire with no perceptible lag, no distortion, and no fluctuation. When changing audio code, prefer firing the sound *before* DOM work, keep Tone's `lookAhead` minimal (currently 25 ms in `bloops.html`), and don't add unconditional time cushions to interactive triggers — guards for cold-start (suspended `AudioContext`) are fine but should not penalize the warm path. If a change risks audio glitches (under-runs, stagger between voices, dropouts), test it in the browser before declaring done.

3. **Keep the audio signal-flow diagram current.** `README.md` (section *"Bloops — audio signal flow"*) holds an ASCII diagram of how notes route from each source through the two entry buses (`globalSendTap` vs. the per-lane bus) to the shared master chain and speakers. Whenever you change audio routing — `playNote()` destination resolution, `globalSendTap`, `getLaneBus()`, the FX send/return wiring, or the master chain order — **update that diagram in the same change** so it never drifts from the code. Prefer one global routing rule over per-mode volume/FX hacks; if modes sound inconsistent, fix it at the bus/master level (and re-draw the diagram) rather than scaling individual call sites.

## Record learnings here (avoid repeat debug churn)

When a bug or behavior takes more than a couple of attempts to figure out — especially anything non-obvious about this codebase (CSS that overrides JS, event/lifecycle ordering, audio scheduling/lookahead, save/load quirks, the Bloom engine, view-mode rules) — **add a short note to the "Gotchas / hard-won learnings" list below in the same change**, so the next task doesn't rediscover it. Keep each entry one or two lines: the symptom and the fix/cause. Check this list before deep-debugging a UI/audio/state issue.

### Gotchas / hard-won learnings
- **Body-attached overlays get force-hidden by view-mode CSS.** Rules like `body.<mode> > *:not(#mix-view):not(.ctx-menu) { display:none !important }` (and the `tracks-fullscreen` family) hide any element appended to `<body>` that isn't exempt. A selector `!important` beats a normal `.open{display:block}` rule, so toggling a class won't show it. Fix: set `display` via **inline `!important`** (`el.style.setProperty('display','block','important')`); clear on close. Diagnose with a toast of `getComputedStyle(el).display` + `document.body.className`.
- **`position:fixed` overlays drift** because the app uses `backdrop-filter` on some ancestors, which makes one a *containing block* — so `getBoundingClientRect`-derived `top/left` are offset (and shift with scroll). Fix: **measure-and-correct** — set the overlay to `0,0`, read its rect (= the containing-block offset), then subtract it from the desired viewport coords.
- **Live-readout chips must not be `<button>` / focusable.** The notes readout repaints on a timer, so a `click` (which needs mousedown+mouseup on the *same* element) gets dropped when the element is recreated between press and release; and focusing a control inside an `aria-hidden` line is blocked. Fix: render chips as non-focusable `<span role="button">` and act on **`pointerdown`**, not click.
- **Bloom edits land one iteration late** unless you account for the ~1.4 s lookahead: the next iteration's voices are already scheduled with old values. To apply on the next iteration, cancel the layer's scheduled-ahead voices and re-anchor/re-emit. Use `cancelBloomFutureVoices(key, fromAt)` (voices are tagged with `_ak`/`_akAt`) to drop *only* the next iteration so the current one finishes intact.
- **Silent/dry Bloom render** already exists: set `window._ambSilentCapture = true` (with a capture sink) and `playNote` captures without sounding — reuse it instead of adding a dry-run path.
- **Always-on per-layer DSP glitches modest Bloom stacks.** Continuous DSP built for every layer even when inactive (it ran at flat/0 settings) drains the audio render thread and glitches/cuts out with only a handful of layers. This is why Distortion/Delay are spliced in LAZILY by `_ambApplyLayerFx` only when `mix > 0`, and why the per-layer **EQ3 is lazy too** (`_ambApplyEq` inserts an EQ3 between vcf→vca only when a band ≠ 0; the meter FFT analyser stays on the VCA so meters still work). Rule: don't add per-layer nodes that run unconditionally — gate them on the feature being engaged, or a 5-layer project chokes. Even *engaged* heavy DSP needs scaling: per-layer **Distortion oversampling** is adaptive (`_ambDistOversample`/`_ambNormalizeDistOversample`) — 1 dist layer = 4x, 2-3 = 2x, 4+ = none — because 5× 4x-oversampled waveshapers (the "FX on all layers" case) overran the render thread and cut audio out (recovered only by Stop→Play, which tears the FX down).
- **A layer's `level` has two synced controls and ~5 writer paths.** The mixer fader and each layer card's "Level" slider are both shortcuts for the same `layer.level`. Writers: mixer fader (`_ambRenderMixer`), primary `bind()`, extras schema `k==='sl'`, seq `bindInt`, sample `bindInt`. Each calls `_ambSyncLevelUI(E, key, v)` to mirror the value into the OTHER control's slider + readout (it only sets `.value`/text — no synthetic `input` dispatch, so no feedback loop). The mixer fader carries `data-mixkey=<layer key>`; cards are found via their header `[data-phkey]`. If you add another place that mutates `level`, call `_ambSyncLevelUI` there too or the two controls drift.
- **Master Bloom decouples the VIEWED area from the PLAYING area.** `getCfg()` (= `masterAmbient` = `areas[activeIdx]`) is the area the PANEL shows/edits; the ENGINE generates `areas[E._playIdx]` via **`_ambPlayCfg(E)`** (returns getCfg for lane/shape/harness — `E._playIdx` is only set on the master while playing, so the invariant harness is unaffected). The two engine cfg sources are `_ambTick` and `_ambSyncMods` (both use `_ambPlayCfg`); generation reads the cached `E._cfg` which the tick sets from the play area. **So while playing you can click a tab to view/edit ANY area without changing playback** (`_ambSwitchToArea` is view-only when `_masterEng.timer`). Orchestration advances `E._playIdx` (NOT `activeIdx`) and does NOT rebuild the panel. CRITICAL: any UI handler that pushes an edit LIVE to the engine (`_ambSyncOneLayerMod`, the FX/EQ `_ambApplyLayerFx` calls) must be gated by **`_ambLiveApplyOK(E)`** (false when editing a non-playing area) so editing a non-playing area can't disturb the running engine's shared-key chains; `_ambSyncMods`-based edits (mod/spread) are inherently safe since `_ambSyncMods` reads the play area. A "Plot" = one full pass through the area sequence (`E._plotCount`); per-area stochastic `playsRandom` → `_ambEffectivePlays` randomizes the play count per entry (`E._curPlays`).
- **Bloom progressions resolve the chord PER-ONSET (bar-aligned), not per-loop or per-ms.** A "Progression" note source's current chord = `_ambProgCurrentChord(n)`, which reads the module-global `_ambProgStepOverride`. Every emit that can play a prog (bass/run/pedal/drone/bed/motif/texture/arp) sets `_ambProgStepOverride = _ambProgStepAt(E, at)` — the bar-aligned chord step at THAT voice's onset `at` (`floor(barsElapsed/barsPerChord)`, anchored at `_playStartAt`/`_progAnchor`) — right before the pitch pick, then clears it. This replaced the legacy ms `progRateMs` clock (now an inert cfg field) and the per-series-loop model, so a multi-bar layer FOLLOWS the changes within its loop. Rate = `cfg.barsPerChord` (Configure → Bars/chord). All gated on `isProg`, so scale layers stay byte-identical (harness scale configs pinned; `prog-bass`/`prog-global`/`prog-arp` cover the new path). **GLOBAL progression**: `cfg.prog={on,name,chords}` (Configure → Prog); `_ambNotesOf` makes any layer on the default `scale:''` source INHERIT it via `_ambGlobalProg()` (explicit scale/chord/wrap/prog overrides). Activating it auto-snaps free continuous layers (bed/motif/texture/beat-random) to the nearest bar-ratio (`_ambAutoSyncFreeForProg`). **ARP**: its chord is `_chordIdx = _ambProgStepAt(_E, at)`, DECOUPLED from `st._loop` (now purely the structural entry-cursor + queue-sizing index); step-mutes/evolve key off `_chordIdx`. `_ambArpSeriesInfo` still sizes by `st._loop`'s chord — exact for uniform chord sizes, minor queue drift only for mixed sizes.
- **Seamless Bloom AREA transitions = stop-generating + let tails ring + reuse shared chains (NO bus dip, NO full teardown).** `_ambOrchAdvance` hands off between areas gaplessly by: (1) `cancelBloomFutureVoices(key, now)` for each layer — this cancels only **pending** (not-yet-started) voices, so currently-sounding voices ring out (release/reverb tails overlap the incoming area = the gapless feel); (2) swapping the active area + reseeding + `_ambResetClocks` (which does NOT clear `E.mod`); (3) a **soft reconcile** — `_ambBuildMod`/update every key in the new area's `_ambWantSet` (shared keys like `bed` keep their chain so tails are uninterrupted, params just update), and for keys only in the OLD area, `_ambGateRamp(…,0,…,0.5)` then dispose after ~0.8 s (no teardown click). An earlier version dipped the bloom bus to 0 and `_ambTeardownMods()`’d everything — that was glitch-free but audibly gapped; don't reintroduce a dip. **Reconfiguring a SHARED chain while its tail rings POPS** (EQ splice / FX re-route / `_ambSyncTarget` LFO dispose+recreate are graph changes = clicks). Fix (`_ambReconfigSharedQuiet`): mute that layer's OWN dry gate ~6 ms, apply the graph change while muted, unmute — localized so the bus stays seamless. (The reverb send taps the VCA *pre-gate*, so a layer's reverb tail keeps ringing through the gate mute — dry-only areas can have a ~20 ms blip, reverb-bearing areas mask it.) Caveats: `cancelBloomFutureVoices` matches by layer KEY not engine, so a lane Bloom playing the same key simultaneously can lose pending voices at a master area switch; and a purely-percussive area (no tails) can have a ~0.3 s pickup gap (the new area's first voices are at the standard `lead`).
- **The When-cursor cell index is derived, not counted, for phase-anchored layers.** The "When" grid highlight used to ride a wrap-counter seeded to −1 on activation (it assumed the layer goes active during the pre-onset *lead*, so the first onset's wrap bumps it to cell 0). But phase-anchored layers (arp / bass / run / pedal / drone / shape / euclid) go active **AT** the first onset (`now ≥ st.startAt`), not during the lead — so the first onset had no wrap to bump 0 and the cursor trailed the audio by exactly one unit. Fix: in the readout RAF (`_ambSyncBars`/bars loop) compute `cyc = Math.floor((now − st.startAt) / period)` — the SAME absolute index scheduling gates `_ambCondFires(when, c)` by — and use it directly (`el._whenCycle = cyc`). It's self-correcting (no dependence on catching the activation edge). Lead-anchored layers (seq/texture/interval, driven off `E.clocks` future onsets) keep the −1-then-wrap counter. `_ambLayerPeriodSec` for a series arp = the whole **series-loop** length (`totalNotes × interval`), which is exactly what When gates, so `cyc` lines up.
- **The Bloom invariant harness hashes `pan`, and the RNG is one shared sequential stream.** `23-bloom-harness.js` records `[at,freq,durMs,voice,pan,level]` per note, so changing a layer's pan VALUE *or* adding/removing an `_ambRand()` call (which shifts every downstream draw) drifts the baseline. The battery covers the SERIES arp (`withExtra('arp')`) but not the euclid arp, so euclid-only pan changes are free; for harness-covered paths, only override the pan *value* when the feature is engaged and keep the RNG draws identical at the defaults (`space:0`).
- **Per-note pan (`params.pan` → playNote's per-voice `Tone.Panner`) does NOT give a Bloom layer audible stereo spread, and is too costly for a dense layer.** For the Euclid Arp it failed twice: (1) at the default 1 voice `_ambSpacePans(1,…)` returns `[0]` (centre); (2) even with non-zero per-note pan the image stayed centred AND a panner-per-hit on a dense arp glitched the render thread (the per-note `Tone.Panner` → `_ambLayerDest` = the mod-chain input `vcf` apparently doesn't survive to the bus as stereo). Pan must be applied at `e.pan` (the layer's output Panner, `_E.mod[key].pan.pan` — the one node confirmed to reach the bus in stereo). **But do NOT schedule it PER HIT** (`setValueAtTime`/`setTargetAtTime` per note): that accumulates automation events on a *persistent* param, a slow build-up that glitches and only clears on Stop→Play (disposing the node) — and turning Spread off mid-play does NOT clear it, so it looks like "glitches regardless of spread." **Audio-safe fix: one persistent `Tone.LFO` per arp layer driving `e.pan.pan`** (`_ambArpSpreadLfo`): bounded (a single node, ZERO per-note scheduling → no timeline growth), click-free, width = Spread, ~2 cycles/bar so consecutive notes land at different points (reads as spread). Disengage/recentre in Pan mode or at `space 0`; tear it down in `_ambTeardownMod` (before disposing `e.pan`) and in the series-arp path (`_ambEmitArpWindow`) so a euclid→series switch drops it. **General rule: never schedule per-note/per-hit automation on a long-lived AudioParam — use a connected LFO/Signal, or per-voice nodes that get disposed.** (The euclid Beat still uses per-note pan via `vpan`; same caveat applies if it ever needs spread.) Fan by SLOT position (`_ambSpacePans(steps, space)[slot]`) so a single voice sweeps L→R with the rhythm. Only when `panMode !== 'pan' && space > 0`; otherwise leave `e.pan` to `_ambApplyLayerPan`. Caveats: `e.pan` is shared by the layer so voices can't pan independently (slot sweep only), the reverb send taps pre-pan (wet stays centred), and `_ambApplyLayerPan` (run on any mod-sync) sets `e.pan.value=0` — momentarily cancelling the scheduled sweep until the next tick re-schedules. The automation timeline grows over a continuous play but resets on Stop/Play (mod teardown).
- **Bloom layer PAN is on a per-layer `Panner` node, not (only) per-voice.** Bloom applies pan at note-onset, so a pan *ramp* via the layer's `space` would only re-pan per note (coarse/inaudible for sustained or sparse layers). Fix: the mod chain has a `pan` Panner (`vca → gate → pan → [FX] → bus`). In **Pan** mode `_ambLayerPan`/`_ambLayerPans` return 0 (per-voice centred) and the Panner holds the position — set from `space` by `_ambApplyLayerPan`, and driven CONTINUOUSLY by the pan ramp (`_ambRampResolve` special-cases `space` → pushes to `e.pan.pan`). In **Spread** mode the Panner stays centred and per-voice pans fan the width. So a "pan ramp" only sweeps the image in **Pan** mode; in Spread mode it animates width.
- **Bloom's first iteration is dropped on a cold-start play** (symptom: "no notes for several iterations, especially Bed"). Pressing Play resumes the AudioContext *asynchronously*; if `_ambTick` SCHEDULES while the context is still `suspended`, its voices anchor to a frozen clock and the first onset is dropped on resume — worst on Bed (long iteration = seconds of silence). Fix: in `_ambStartGenerator` start the interval + flip `cfg.playing`/the Play button IMMEDIATELY (do NOT defer the whole start — that left the button on ▶ and added a ~1 s delay), and instead gate only the SCHEDULING: `_ambTick` returns early while `Tone.getContext().rawContext.state !== 'running'`. The interval keeps ticking, so the first running tick (≤150 ms after resume) anchors cleanly; `_ambStartGenerator` also kicks one extra tick from `Tone.start().then(...)` so cold starts don't wait a full interval. **CRITICAL: this scheduling guard must apply at COLD START ONLY** (gate it behind `E._everRan`, set true on the first 'running' tick). If it keeps blocking whenever the context isn't 'running', a mid-playback context dip (tab backgrounded / power blip) starves the schedule and turns a momentary glitch into a full cut-out — the keep-alive watchdog + 1.4 s lookahead already cover steady-state, so never block once it has run.
- **EVERY area transition is a SECTION transition (depart + per-layer fade), NOT the seamless reconcile, and NOT gated on Bar Lock.** `_ambOrchAdvance` departs every outgoing layer via `_ambDepartLayer` — its chain is moved to a unique `key#depN` key, gate-faded to 0 over that layer's `areaFadeMs`, then (after the fade) its sounding voices are STOPPED via `stopBloomVoicesBefore(key, at)` and the chain disposed. TWO non-obvious musts: (1) the stop/teardown `setTimeout` is timed from the FUTURE boundary `at` (`(at-Tone.now())*1000 + fadeMs + 60`), NOT the advance-call time — the advance fires ~0.6 s BEFORE the boundary, so a delay measured from it disposed the chain BEFORE the gate-fade (scheduled at `at`) even ran, killing the fade. (2) The gate-fade only silences GATE-ROUTED voices; looping sample/pad voices (`_activePadVoices`, and one-shot sample voices) would ring on + the new area retriggers more → "samples run over and get louder". `stopBloomVoicesBefore(key, beforeAt)` (04-instruments-samples.js) stops active+pending synth/sample/pad voices with `_ak===key && _akAt < beforeAt` (outgoing area only; incoming voices start ≥ boundary so they're spared) — pad voices are now `_ak`-tagged at creation so they're targetable. So departure = gate-fade (the audible fade) + voice-stop (the guarantee) — while the incoming area builds FRESH chains on the real keys. Area fade applies ONLY to **Free** layers (the slider, default 250 ms, normalized in `mig`); **synced / bar-native layers HARD-CUT at the boundary** (fade forced to 0 in `_ambAreaFadeMap` via `_ambIsCapturable`) — they're loop/structure, so the slider doesn't apply. EVERY area transition uses this depart path now (NOT gated on barLock — barLock only governs whether layers LOOP); the old shared-chain reconcile (`_ambReconfigSharedQuiet` + old-only gate-ramp) is no longer used by the advance. DON'T re-gate the depart on `barLock`/`_barLockLoopBars` — Free layers have `_barLockLoopBars=0` and would fall through and never fade. Reason for departing to a temp key: a layer has ONE chain per key, so to overlap old-fading + new-fresh for the same key you must move the old one aside first. The advance times off `_barGridAnchor` (the lead grid, reset per advance), not `_orchStartAt`, AND fires EARLY with a GENEROUS margin (~0.6 s lead) so the advance always fires BEFORE the boundary even under tick jank (the tick handler can take 50–70 ms). It passes the precise boundary to `_ambOrchAdvance(E, boundaryAt)`, which sets `at = boundaryAt > now ? boundaryAt : now` and cancels pending voices + starts each layer's fade AT `at`. CRITICAL: the lead must be big enough that `at` is the FUTURE boundary, not a late `now` — if the advance fires AFTER the boundary, `at` falls back to `now` and the outgoing SYNCED layers (which can't cancel already-started voices) bleed past the boundary = overflow. A too-small lead (0.3 s) caused exactly that under jank. The incoming area is anchored to the boundary (`E._barGridAnchor = at`), so its synced layers' first onsets land on the next bar AT the boundary, not before it (overlap) — both the outgoing cut and the incoming start sit on the boundary.

- **Sample/pad voices register as ACTIVE immediately (even when scheduled ahead) — so future-scheduled ones live in the ACTIVE sets, not the pending sets.** This bit the area-transition departure: `cancelBloomFutureVoices(key, fromAt)` only scanned `_pendingVoices`/`_pendingSampleVoices`, so a sample voice scheduled to start AT/AFTER the boundary (in `_activeSampleVoices`, `_akAt >= fromAt`) was neither cancelled (not pending) nor stopped by `stopBloomVoicesBefore(key, beforeAt)` (which only stops `_akAt < beforeAt`). It then started in the NEXT area and piled up across transitions = "samples run over and get louder". Fix: `cancelBloomFutureVoices` now ALSO scans `_activeVoices`/`_activeSampleVoices`/`_activePadVoices` for `_akAt >= fromAt` (not-yet-sounded → safe to cancel, no click). Synth voices mostly defer to `_pendingVoices` so they were already caught; SAMPLES are the ones that leak. Diagnose voice leaks with the `_activeSampleVoices.size` / `_activePadVoices.size` totals.

- **A LOCKED layer's notes render as an EDITABLE piano-roll canvas (`lockroll` mode), not chips.** `_ambUpdateNotesLive` switches a `.ambient-notes-live` line to a `<canvas class="ambient-np-roll ambient-np-lockroll">` when `_ambLockedNotes` returns notes (which it does for ANY frozen layer, incl. Bar-Lock `_barLock` — so Bar Lock auto-exposes editing = item "#3B" needs no extra code). `_ambDrawLockedRoll` draws the loop STATIC over its period (`_ambLockMeta` gives list/offKey/durKey/P) with a playhead from `E.freeze[key].anchor`; `_ambWireLockedRoll` hit-tests a pointerdown to a note index → `_ambOpenNoteEditor` (click block = edit) or `_ambAddNoteAt` (click empty = add there). GOTCHA: the editor-dismiss check in `_ambUpdateNotesLive` must NOT require a `.ambient-np-note` chip (there are none in roll mode) — it falls back to validating `_ambNoteEd.index` against `_ambLockedNotes(...).list`, or the popover closes itself one frame after opening. Drawn every frame by `_ambDrawRolls` (canvas created once on mode-change, redrawn in the viz rAF). Pure UI/edit (edits apply via `_ambReemitLockedNext`), so the note-generation harness is unaffected.

- **The "Riff" layer is internally the `run` type (display rename only).** `_AMB_LAYER_SCHEMA.run.label` and the Add-layer menu say "Riff", but the config key, `_ambEmitRun`, `E.runPhase`, the period/notesOf maps, and saved projects all still use `'run'`. Don't rename the TYPE — it would orphan saved projects and break the harness. Riff's controls map: # notes = Density×Bars, # bars = Bars, rests = Rests, note-length distribution = **Len var** (`lenVary`, via the gated `_ambVaryLen` → no rng draw at 0, so the default stays harness-identical).

- **The Bloom tick captures `cfg` ONCE at the top; the orch-advance runs MID-tick and swaps the play area — so after firing the advance the tick MUST `return`, or it re-generates the OUTGOING area with the stale `cfg`.** `_ambTick` reads `cfg = _ambPlayCfg(E)` (the play area) near the top, then the orch boundary check (`now >= boundary - lead`) calls `_ambOrchAdvance` which swaps `E._playIdx` to the new area, cancels the outgoing layers' voices, AND generates the new area (its own `_ambTick`). If the original tick then CONTINUES into its layer-generation section, it emits the OLD area's layers again using the captured (stale) `cfg` — those voices are scheduled INTO the new area's time and AFTER the cancel ran, so nothing cancels them → they bleed into the next area (symptom: "bass/pedal play in the next area even though it has no bass/pedal layer"). Fix: `return` immediately after `_ambOrchAdvance` (the advance already generated the new area; `_orchSwitching` is reset at the end of the advance so the next tick proceeds normally). This is why the depart/cancel/stop machinery all looked correct in isolation but layers still overflowed — the re-emission happened downstream of all of it.

## Architecture

This is a browser-based music player that streams audio files from Google Drive. It is a single-page app served by a minimal Express static server (`server.js`).

### Module structure

Four vanilla JS modules are loaded in order via `<script>` tags in `index.html`:

1. **`js/google-drive-api.js`** — `GoogleDriveAPI` class. Initializes `gapi` and Google Identity Services (GIS) OAuth client, handles sign-in/sign-out, searches Drive folders, paginates through audio files, and fetches artist metadata.
2. **`js/music-player.js`** — `MusicPlayer` class. Wraps the HTML5 `<audio>` element, manages blob URL lifecycle (creates on play, revokes on next load), and handles keyboard shortcuts (Space, Arrow keys).
3. **`js/playlist-manager.js`** — `PlaylistManager` class. Owns the track array, renders the playlist DOM, handles drag-and-drop and touch reordering, and implements shuffle (Fisher-Yates, preserves original order for restore).
4. **`js/app.js`** — Wires everything together, manages auth UI state, loads playlist options from a Drive file at `bloops/playlists`, and shows loading/toast feedback.

### Event-driven communication

Modules communicate via custom DOM events (no external event bus):

| Event | Source | Consumer |
|---|---|---|
| `authStatusChanged` | `GoogleDriveAPI` | `app.js` (updates UI) |
| `trackLoaded` | `MusicPlayer` | `app.js` |
| `trackEnded` | `MusicPlayer` | `PlaylistManager` (advance track) |
| `requestNextTrack` | `MusicPlayer` | `PlaylistManager` |
| `requestPreviousTrack` | `MusicPlayer` | `PlaylistManager` |

### Key data flows

**Auth:** User clicks connect → GIS OAuth popup → access token stored in `GoogleDriveAPI` instance → `authStatusChanged` fires → UI updates.

**Loading music:** Folder name selected/searched → `driveAPI.searchFolders()` → `driveAPI.getMusicFilesFromFolders()` (paginated) → optionally `driveAPI.fetchArtistName()` from a special "Artist name" file in the folder → `playlist.setTracks()` renders list → first track auto-loads.

**Playback:** `PlaylistManager.playTrack(index)` → `MusicPlayer.loadTrack()` → fetch audio blob from Drive with `Authorization: Bearer <token>` header → blob converted to object URL → set as `<audio>` src → play.

### Google Drive folder convention

The app expects:
- A `bloops/` folder at the root of My Drive with subfolders `bloops/effects`, `bloops/projects`, `bloops/exports`. The Bloops sign-in init creates these on first run with one consolidated confirmation prompt.
- A text file at `bloops/playlists` listing folder paths (one per line) that the Player turns into playlists. Paths use `/` for parent/child nesting (e.g. `bloops/exports`). The init step seeds this file with `bloops/exports`.
- Audio folders optionally containing a file named **"Artist name"** whose first line is used as the artist, and an image named **"Artwork"** used as album art.
- Supported audio formats: `mp3, wav, flac, aac, ogg, m4a, opus`.

### Google API credentials

`CLIENT_ID` and `API_KEY` are hardcoded in `js/google-drive-api.js`. To use a different Google Cloud project, replace those constants and update the authorized JavaScript origins in the OAuth client configuration. The app uses the `drive.readonly` scope.

### Deployment

`deploy.sh` stages the project, installs production dependencies, and uploads via SFTP using `lftp`. See `pre_deploy.txt` for GoDaddy cPanel prerequisites (Node.js app setup, SFTP credentials).
