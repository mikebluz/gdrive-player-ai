# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## UI guidance

Apply these rules to every UI change:

1. **No horizontal scrolling.** If a row's contents don't fit, either (a) shrink elements in that row to fit (`min-width: 0`, `flex-shrink: 1`, narrower padding/font on small viewports via media queries), or (b) wrap to a new row and resize the new row's contents to fit. Never solve overflow with `overflow-x: auto/scroll`.
   - **Lane step strips (`.lane-chips`) are a BAR GRID** (this replaced the old single-row scrolling timeline â€” the prior carve-out is gone). Each `.lane-chips` is a CSS grid `repeat(_BAR_SUBCELLS, 1fr)` where **1 row = 1 bar** = `_BAR_SUBCELLS` (32) sub-cells, fitting the strip width with **no horizontal scroll**. A step spans `round(stepLengthFactor Ă— 32)` cells (one whole note = a full bar/row); a step crossing a bar line **splits into continuation segments** (`.cont-start` head + `.cont-mid`/`.cont-end` clones, marked `.cont-segment`) via `_barGridPlan`. Continuation segments are visual-only (no handlers, excluded from index-based selectors by `:not(.cont-segment)`). The old horizontal sweeping playhead cursor (`_followChipInLane`/`_positionCursorsAtTick`) is disabled â€” playback shows via the per-chip `.active-loop` highlight. Open follow-ups: triplets/odd meters (32 isn't divisible by 3, so they round to the nearest cell) and a proper grid playhead.

2. **Audio playback must be smooth and immediate.** Every user-triggered sound (cell press, wrap audition, sequence/loop playback, REST tap, sample preview, recording playback, etc.) should fire with no perceptible lag, no distortion, and no fluctuation. When changing audio code, prefer firing the sound *before* DOM work, keep Tone's `lookAhead` minimal (currently 25 ms in `bloops.html`), and don't add unconditional time cushions to interactive triggers â€” guards for cold-start (suspended `AudioContext`) are fine but should not penalize the warm path. If a change risks audio glitches (under-runs, stagger between voices, dropouts), test it in the browser before declaring done.

3. **Keep the audio signal-flow diagram current.** `README.md` (section *"Bloops â€” audio signal flow"*) holds an ASCII diagram of how notes route from each source through the two entry buses (`globalSendTap` vs. the per-lane bus) to the shared master chain and speakers. Whenever you change audio routing â€” `playNote()` destination resolution, `globalSendTap`, `getLaneBus()`, the FX send/return wiring, or the master chain order â€” **update that diagram in the same change** so it never drifts from the code. Prefer one global routing rule over per-mode volume/FX hacks; if modes sound inconsistent, fix it at the bus/master level (and re-draw the diagram) rather than scaling individual call sites.

## Record learnings here (avoid repeat debug churn)

When a bug or behavior takes more than a couple of attempts to figure out â€” especially anything non-obvious about this codebase (CSS that overrides JS, event/lifecycle ordering, audio scheduling/lookahead, save/load quirks, the Bloom engine, view-mode rules) â€” **add a short note to the "Gotchas / hard-won learnings" list below in the same change**, so the next task doesn't rediscover it. Keep each entry one or two lines: the symptom and the fix/cause. Check this list before deep-debugging a UI/audio/state issue.

### Gotchas / hard-won learnings
- **Wrap-ensemble = a wrap with PER-NOTE TONES, voiced by Bloom; the LAYER MODE supplies the shape.** Wraps always gave Bloom only PITCH (`intervals`+`root`); the wrap-ensemble adds per-degree TONE. `_ambPublishWrap` captures `wrap.tones[]` (parallel to `intervals`) from the wrap's chord notes that carry `toneOverride`+`sound` (set in the Seed wrap step-editor â€” already existed). `_ambDegreeToneAt(deg, src)` returns a wrap's per-degree tone (null otherwise). Two delivery mechanisms: (1) SINGLE-note emits read a side-channel â€” `_ambDegreeFreq` stamps `_ambLastDegTone` on every pick, and bass/run/motif/pedal/texture do `if (_ambLastDegTone) params.type = _ambLastDegTone` right after the pick (works only because the pick immediately precedes the param build, no intervening pick). (2) MULTI-pick emits read INLINE (the side-channel would be stale): bed `_ambPickVoicing` chaos path builds `{f,t}` pairs, sorts by freq, and hangs a parallel `freqs._tones` on the returned array which `_ambEmitBed` applies per voice (= the **Stack** shape; structured Chords/Monk modes use chord FORMS not wrap degrees, so they don't carry tones); arp series uses `_ambDegreeToneAt(d, notes)` with the interval index `d`; arp euclid stores `tone` on each precomputed hit at build (its `deg` isn't in scope at play). **The shape is the layer**: single-note layers â†’ Run (a melodic line, each note its own timbre), bed chord â†’ Stack. **Gated/harness-safe:** every site applies the tone ONLY when non-null, so a non-wrap source (or a wrap with no per-note tones) is byte-identical to before â€” the note stream (and the invariant harness, which doesn't hash tone) is unchanged. NOT covered: the bed STRUCTURED modes (form-based, not degree-based).
- **Body-attached overlays get force-hidden by view-mode CSS.** Rules like `body.<mode> > *:not(#mix-view):not(.ctx-menu) { display:none !important }` (and the `tracks-fullscreen` family) hide any element appended to `<body>` that isn't exempt. A selector `!important` beats a normal `.open{display:block}` rule, so toggling a class won't show it. Fix: set `display` via **inline `!important`** (`el.style.setProperty('display','block','important')`); clear on close. Diagnose with a toast of `getComputedStyle(el).display` + `document.body.className`.
- **`position:fixed` overlays drift** because the app uses `backdrop-filter` on some ancestors, which makes one a *containing block* â€” so `getBoundingClientRect`-derived `top/left` are offset (and shift with scroll). Fix: **measure-and-correct** â€” set the overlay to `0,0`, read its rect (= the containing-block offset), then subtract it from the desired viewport coords.
- **Live-readout chips must not be `<button>` / focusable.** The notes readout repaints on a timer, so a `click` (which needs mousedown+mouseup on the *same* element) gets dropped when the element is recreated between press and release; and focusing a control inside an `aria-hidden` line is blocked. Fix: render chips as non-focusable `<span role="button">` and act on **`pointerdown`**, not click.
- **Bloom edits land one iteration late** unless you account for the ~1.4 s lookahead: the next iteration's voices are already scheduled with old values. To apply on the next iteration, cancel the layer's scheduled-ahead voices and re-anchor/re-emit. Use `cancelBloomFutureVoices(key, fromAt)` (voices are tagged with `_ak`/`_akAt`) to drop *only* the next iteration so the current one finishes intact.
- **Silent/dry Bloom render** already exists: set `window._ambSilentCapture = true` (with a capture sink) and `playNote` captures without sounding â€” reuse it instead of adding a dry-run path.
- **Always-on per-layer DSP glitches modest Bloom stacks.** Continuous DSP built for every layer even when inactive (it ran at flat/0 settings) drains the audio render thread and glitches/cuts out with only a handful of layers. This is why Distortion/Delay are spliced in LAZILY by `_ambApplyLayerFx` only when `mix > 0`, and why the per-layer **EQ3 is lazy too** (`_ambApplyEq` inserts an EQ3 between vcfâ†’vca only when a band â‰  0; the meter FFT analyser stays on the VCA so meters still work). Rule: don't add per-layer nodes that run unconditionally â€” gate them on the feature being engaged, or a 5-layer project chokes. Even *engaged* heavy DSP needs scaling: per-layer **Distortion oversampling** is adaptive (`_ambDistOversample`/`_ambNormalizeDistOversample`) â€” 1 dist layer = 4x, 2-3 = 2x, 4+ = none â€” because 5Ă— 4x-oversampled waveshapers (the "FX on all layers" case) overran the render thread and cut audio out (recovered only by Stopâ†’Play, which tears the FX down).
- **A layer's `level` has two synced controls and ~5 writer paths.** The mixer fader and each layer card's "Level" slider are both shortcuts for the same `layer.level`. Writers: mixer fader (`_ambRenderMixer`), primary `bind()`, extras schema `k==='sl'`, seq `bindInt`, sample `bindInt`. Each calls `_ambSyncLevelUI(E, key, v)` to mirror the value into the OTHER control's slider + readout (it only sets `.value`/text â€” no synthetic `input` dispatch, so no feedback loop). The mixer fader carries `data-mixkey=<layer key>`; cards are found via their header `[data-phkey]`. If you add another place that mutates `level`, call `_ambSyncLevelUI` there too or the two controls drift.
- **Bloom Level is a CONTINUOUS per-layer GAIN node (`e.levelGain`), NOT a per-note volume.** Chain: `vcf â†’ vca â†’ levelGain â†’ gate â†’ pan` (`levelGain` spliced in `_ambBuildMod`). `_ambApplyLevel` is now a NO-OP pass-through; the level is applied as a gain via `_ambLevelGain(level)` (0â†’silent, 70â†’unity, 100â†’Ă—2), written by `_ambUpdateMod` (build/sync), `_ambSyncLevelUI` (every manual writer â€” real-time fader), and the ramp's `key==='level'` `set` (real-time sweep). **Why:** per-note level only affected NEW notes, so on a sustained/overlapping pad the held tails masked it ("the level ramp does nothing"); a layer gain sweeps the WHOLE layer, tails included, in real time. The reverb send now taps `levelGain` (still pre-gate â†’ scales with level AND tails ring through a gate mute). **Harness-safe:** at the default level 70 the no-op equals the old curve exactly, and no harness config sets a non-default level, so the invariant battery is byte-identical. If you add a new `level` writer, push to `e.levelGain.gain` (via `_ambSyncLevelUI`) or held voices won't follow.
- **The TRANCE GATE is a bar-synced step pattern on a dedicated chain node (`e.tgGate`), driven by a `Tone.Signal` (`e.tgSig`).** Chain: `vca â†’ levelGain â†’ tgGate â†’ gate` (tgGate spliced in `_ambBuildMod`, pre-gate so the reverb â€” tapped off `levelGain` â€” fills the chopped gaps). Config is per-layer in the FX object: `tg:{on,steps,pattern[],depth,edge}` (`_ambDefaultFx`/`_ambNormalizeFx`; `steps` = steps/bar = pattern length). The driver `_ambScheduleTg` (called per layer from `_ambScheduleStochastic` each tick) schedules the pattern AHEAD onto `tgSig`: **`tgGate.gain` intrinsic is 0 and the connected `tgSig` sets it** (a connected signal SUMS with the param, so intrinsic must be 0, not 1). When `tg.on` is false it disposes `tgSig` and parks `tgGate.gain=1` (passthrough). The grid is anchored to `E._playStartAt` at the cfg `bpm` (`barSec=(60/bpm)*4`) so it locks to the bar. `tgSig` is disposed on stop/teardown so the automation timeline can't accumulate across plays (the codebase's rule â€” a dedicated disposable Signal, never per-step automation on a persistent shared param). UI: a "Trance Gate" subsection (On/Steps/Depth/Edge + a step grid reusing `.ambient-slice-grid`/`.ambient-slice-cell`) that lives in **`_ambModUi`, grouped under the VCA mod** (both modulate gain â€” VCA = smooth LFO, gate = step pattern); its element ids are still `fx-tg-*` (unchanged when it moved out of `_ambFxUi`), so the shared `_ambWireTg(E, el, getL, persist)` â€” called from all four FX wire clusters (seq/sample/extras/primary) â€” still binds it by id. Harness-safe: `tg.on` defaults 0 and the gate only touches the gain node, never note generation.
- **Portamento (per-layer pitch glide) is GATED and only covers the OSCILLATOR voice path.** A layer's `portamento` (ms, shared default 0 via `_ambNormalizeSpread`) is read by `_ambApplyAdsr` (the shared params builder for all 13 Bloom emits), which â€” only when `portamento > 0` â€” sets `p.glideMs` + `p.glideLayer = inst`. In `playNote` (04, the wavetable/oscillator branch), when `glideMs > 0` each `Tone.Oscillator.frequency` is ramped from the layer's previous note freq (tracked on `glideLayer._glidePrev`) to the target over `glideMs`. **Gated** so the default path (and the invariant harness, which doesn't hash glide anyway) is byte-identical. **COVERAGE (all wired):** the glide is applied in THREE `playNote` voice paths â€” (a) oscillator/wavetable (ramp each `Tone.Oscillator.frequency`), (b) the `.frequency` synths FM/AM/Duo/Mono/Membrane/Metal (ramp `synth.frequency` after `triggerAttackRelease`), and (c) **samples** (ramp `source.playbackRate`, tracked on `glideLayer._glidePrevRate` â€” note the slice window is computed at the target rate, so a brief glide can drift the slice slightly). Pluck/Noise don't glide (no sustained pitch). The `portamento` property is on EVERY Bloom layer (`_ambNormalizeFx`) with a UI slider: primary layers (literal `sl`), seq+sample (`_ambSl`), extras (`_AMB_LAYER_SCHEMA` `['sl','portamento',â€¦]`). **Seed (grid lanes)** are wired too: `lane.portamento` (`_makeLane`), injected into the scheduler's SINGLE-note step path (`07-playback-scheduler.js` â€” chords skip it, they'd share one `_glidePrev`), `glideLayer` = the lane object, control = a slider in `_showLaneMenu`.
- **Master Bloom decouples the VIEWED area from the PLAYING area.** `getCfg()` (= `masterAmbient` = `areas[activeIdx]`) is the area the PANEL shows/edits; the ENGINE generates `areas[E._playIdx]` via **`_ambPlayCfg(E)`** (returns getCfg for lane/shape/harness â€” `E._playIdx` is only set on the master while playing, so the invariant harness is unaffected). The two engine cfg sources are `_ambTick` and `_ambSyncMods` (both use `_ambPlayCfg`); generation reads the cached `E._cfg` which the tick sets from the play area. **So while playing you can click a tab to view/edit ANY area without changing playback** (`_ambSwitchToArea` is view-only when `_masterEng.timer`). Orchestration advances `E._playIdx` (NOT `activeIdx`) and does NOT rebuild the panel. CRITICAL: any UI handler that pushes an edit LIVE to the engine (`_ambSyncOneLayerMod`, the FX/EQ `_ambApplyLayerFx` calls) must be gated by **`_ambLiveApplyOK(E)`** (false when editing a non-playing area) so editing a non-playing area can't disturb the running engine's shared-key chains; `_ambSyncMods`-based edits (mod/spread) are inherently safe since `_ambSyncMods` reads the play area. A "Plot" = one full pass through the area sequence (`E._plotCount`); per-area stochastic `playsRandom` â†’ `_ambEffectivePlays` randomizes the play count per entry (`E._curPlays`).
- **Bloom progressions resolve the chord PER-ONSET (bar-aligned), not per-loop or per-ms.** A "Progression" note source's current chord = `_ambProgCurrentChord(n)`, which reads the module-global `_ambProgStepOverride`. Every emit that can play a prog (bass/run/pedal/drone/bed/motif/texture/arp) sets `_ambProgStepOverride = _ambProgStepAt(E, at)` â€” the bar-aligned chord step at THAT voice's onset `at` (`floor(barsElapsed/barsPerChord)`, anchored at `_playStartAt`/`_progAnchor`) â€” right before the pitch pick, then clears it. This replaced the legacy ms `progRateMs` clock (now an inert cfg field) and the per-series-loop model, so a multi-bar layer FOLLOWS the changes within its loop. Rate = `cfg.barsPerChord` (Configure â†’ Bars/chord). All gated on `isProg`, so scale layers stay byte-identical (harness scale configs pinned; `prog-bass`/`prog-global`/`prog-arp` cover the new path). **GLOBAL progression**: `cfg.prog={on,name,chords}` (Configure â†’ Prog); `_ambNotesOf` makes any layer on the default `scale:''` source INHERIT it via `_ambGlobalProg()` (explicit scale/chord/wrap/prog overrides). Activating it auto-snaps free continuous layers (bed/motif/texture/beat-random) to the nearest bar-ratio (`_ambAutoSyncFreeForProg`). **ARP**: its chord is `_chordIdx = _ambProgStepAt(_E, at)`, DECOUPLED from `st._loop` (now purely the structural entry-cursor + queue-sizing index); step-mutes/evolve key off `_chordIdx`. `_ambArpSeriesInfo` still sizes by `st._loop`'s chord â€” exact for uniform chord sizes, minor queue drift only for mixed sizes.
- **Seamless Bloom AREA transitions = stop-generating + let tails ring + reuse shared chains (NO bus dip, NO full teardown).** `_ambOrchAdvance` hands off between areas gaplessly by: (1) `cancelBloomFutureVoices(key, now)` for each layer â€” this cancels only **pending** (not-yet-started) voices, so currently-sounding voices ring out (release/reverb tails overlap the incoming area = the gapless feel); (2) swapping the active area + reseeding + `_ambResetClocks` (which does NOT clear `E.mod`); (3) a **soft reconcile** â€” `_ambBuildMod`/update every key in the new area's `_ambWantSet` (shared keys like `bed` keep their chain so tails are uninterrupted, params just update), and for keys only in the OLD area, `_ambGateRamp(â€¦,0,â€¦,0.5)` then dispose after ~0.8 s (no teardown click). An earlier version dipped the bloom bus to 0 and `_ambTeardownMods()`â€™d everything â€” that was glitch-free but audibly gapped; don't reintroduce a dip. **Reconfiguring a SHARED chain while its tail rings POPS** (EQ splice / FX re-route / `_ambSyncTarget` LFO dispose+recreate are graph changes = clicks). Fix (`_ambReconfigSharedQuiet`): mute that layer's OWN dry gate ~6 ms, apply the graph change while muted, unmute â€” localized so the bus stays seamless. (The reverb send taps the VCA *pre-gate*, so a layer's reverb tail keeps ringing through the gate mute â€” dry-only areas can have a ~20 ms blip, reverb-bearing areas mask it.) Caveats: `cancelBloomFutureVoices` matches by layer KEY not engine, so a lane Bloom playing the same key simultaneously can lose pending voices at a master area switch; and a purely-percussive area (no tails) can have a ~0.3 s pickup gap (the new area's first voices are at the standard `lead`).
- **The When-cursor cell index is derived, not counted, for phase-anchored layers.** The "When" grid highlight used to ride a wrap-counter seeded to âˆ’1 on activation (it assumed the layer goes active during the pre-onset *lead*, so the first onset's wrap bumps it to cell 0). But phase-anchored layers (arp / bass / run / pedal / drone / shape / euclid) go active **AT** the first onset (`now â‰¥ st.startAt`), not during the lead â€” so the first onset had no wrap to bump 0 and the cursor trailed the audio by exactly one unit. Fix: in the readout RAF (`_ambSyncBars`/bars loop) compute `cyc = Math.floor((now âˆ’ st.startAt) / period)` â€” the SAME absolute index scheduling gates `_ambCondFires(when, c)` by â€” and use it directly (`el._whenCycle = cyc`). It's self-correcting (no dependence on catching the activation edge). Lead-anchored layers (seq/texture/interval, driven off `E.clocks` future onsets) keep the âˆ’1-then-wrap counter. `_ambLayerPeriodSec` for a series arp = the whole **series-loop** length (`totalNotes Ă— interval`), which is exactly what When gates, so `cyc` lines up.
- **The Bloom invariant harness hashes `pan`, and the RNG is one shared sequential stream.** `23-bloom-harness.js` records `[at,freq,durMs,voice,pan,level]` per note, so changing a layer's pan VALUE *or* adding/removing an `_ambRand()` call (which shifts every downstream draw) drifts the baseline. The battery covers the SERIES arp (`withExtra('arp')`) but not the euclid arp, so euclid-only pan changes are free; for harness-covered paths, only override the pan *value* when the feature is engaged and keep the RNG draws identical at the defaults (`space:0`).
- **Per-note pan (`params.pan` â†’ playNote's per-voice `Tone.Panner`) does NOT give a Bloom layer audible stereo spread, and is too costly for a dense layer.** For the Euclid Arp it failed twice: (1) at the default 1 voice `_ambSpacePans(1,â€¦)` returns `[0]` (centre); (2) even with non-zero per-note pan the image stayed centred AND a panner-per-hit on a dense arp glitched the render thread (the per-note `Tone.Panner` â†’ `_ambLayerDest` = the mod-chain input `vcf` apparently doesn't survive to the bus as stereo). Pan must be applied at `e.pan` (the layer's output Panner, `_E.mod[key].pan.pan` â€” the one node confirmed to reach the bus in stereo). **But do NOT schedule it PER HIT** (`setValueAtTime`/`setTargetAtTime` per note): that accumulates automation events on a *persistent* param, a slow build-up that glitches and only clears on Stopâ†’Play (disposing the node) â€” and turning Spread off mid-play does NOT clear it, so it looks like "glitches regardless of spread." **Audio-safe fix: one persistent `Tone.LFO` per arp layer driving `e.pan.pan`** (`_ambArpSpreadLfo`): bounded (a single node, ZERO per-note scheduling â†’ no timeline growth), click-free, width = Spread, ~2 cycles/bar so consecutive notes land at different points (reads as spread). Disengage/recentre in Pan mode or at `space 0`; tear it down in `_ambTeardownMod` (before disposing `e.pan`) and in the series-arp path (`_ambEmitArpWindow`) so a euclidâ†’series switch drops it. **General rule: never schedule per-note/per-hit automation on a long-lived AudioParam â€” use a connected LFO/Signal, or per-voice nodes that get disposed.** (The euclid Beat still uses per-note pan via `vpan`; same caveat applies if it ever needs spread.) Fan by SLOT position (`_ambSpacePans(steps, space)[slot]`) so a single voice sweeps Lâ†’R with the rhythm. Only when `panMode !== 'pan' && space > 0`; otherwise leave `e.pan` to `_ambApplyLayerPan`. Caveats: `e.pan` is shared by the layer so voices can't pan independently (slot sweep only), the reverb send taps pre-pan (wet stays centred), and `_ambApplyLayerPan` (run on any mod-sync) sets `e.pan.value=0` â€” momentarily cancelling the scheduled sweep until the next tick re-schedules. The automation timeline grows over a continuous play but resets on Stop/Play (mod teardown).
- **Bloom layer PAN is on a per-layer `Panner` node, not (only) per-voice.** Bloom applies pan at note-onset, so a pan *ramp* via the layer's `space` would only re-pan per note (coarse/inaudible for sustained or sparse layers). Fix: the mod chain has a `pan` Panner (`vca â†’ gate â†’ pan â†’ [FX] â†’ bus`). In **Pan** mode `_ambLayerPan`/`_ambLayerPans` return 0 (per-voice centred) and the Panner holds the position â€” set from `space` by `_ambApplyLayerPan`, and driven CONTINUOUSLY by the pan ramp (`_ambRampResolve` special-cases `space` â†’ pushes to `e.pan.pan`). In **Spread** mode the Panner stays centred and per-voice pans fan the width. So a "pan ramp" only sweeps the image in **Pan** mode; in Spread mode it animates width.
- **Bloom's first iteration is dropped on a cold-start play** (symptom: "no notes for several iterations, especially Bed"). Pressing Play resumes the AudioContext *asynchronously*; if `_ambTick` SCHEDULES while the context is still `suspended`, its voices anchor to a frozen clock and the first onset is dropped on resume â€” worst on Bed (long iteration = seconds of silence). Fix: in `_ambStartGenerator` start the interval + flip `cfg.playing`/the Play button IMMEDIATELY (do NOT defer the whole start â€” that left the button on â–¶ and added a ~1 s delay), and instead gate only the SCHEDULING: `_ambTick` returns early while `Tone.getContext().rawContext.state !== 'running'`. The interval keeps ticking, so the first running tick (â‰¤150 ms after resume) anchors cleanly; `_ambStartGenerator` also kicks one extra tick from `Tone.start().then(...)` so cold starts don't wait a full interval. **CRITICAL: this scheduling guard must apply at COLD START ONLY** (gate it behind `E._everRan`, set true on the first 'running' tick). If it keeps blocking whenever the context isn't 'running', a mid-playback context dip (tab backgrounded / power blip) starves the schedule and turns a momentary glitch into a full cut-out â€” the keep-alive watchdog + 1.4 s lookahead already cover steady-state, so never block once it has run.
- **EVERY area transition is a SECTION transition (depart + per-layer fade), NOT the seamless reconcile, and NOT gated on Bar Lock.** `_ambOrchAdvance` departs every outgoing layer via `_ambDepartLayer` â€” its chain is moved to a unique `key#depN` key, gate-faded to 0 over that layer's `areaFadeMs`, then (after the fade) its sounding voices are STOPPED via `stopBloomVoicesBefore(key, at)` and the chain disposed. TWO non-obvious musts: (1) the stop/teardown `setTimeout` is timed from the FUTURE boundary `at` (`(at-Tone.now())*1000 + fadeMs + 60`), NOT the advance-call time â€” the advance fires ~0.6 s BEFORE the boundary, so a delay measured from it disposed the chain BEFORE the gate-fade (scheduled at `at`) even ran, killing the fade. (2) The gate-fade only silences GATE-ROUTED voices; looping sample/pad voices (`_activePadVoices`, and one-shot sample voices) would ring on + the new area retriggers more â†’ "samples run over and get louder". `stopBloomVoicesBefore(key, beforeAt)` (04-instruments-samples.js) stops active+pending synth/sample/pad voices with `_ak===key && _akAt < beforeAt` (outgoing area only; incoming voices start â‰¥ boundary so they're spared) â€” pad voices are now `_ak`-tagged at creation so they're targetable. So departure = gate-fade (the audible fade) + voice-stop (the guarantee) â€” while the incoming area builds FRESH chains on the real keys. Area fade applies ONLY to **Free** layers (the slider, default 250 ms, normalized in `mig`); **synced / bar-native layers HARD-CUT at the boundary** (fade forced to 0 in `_ambAreaFadeMap` via `_ambIsCapturable`) â€” they're loop/structure, so the slider doesn't apply. EVERY area transition uses this depart path now (NOT gated on barLock â€” barLock only governs whether layers LOOP); the old shared-chain reconcile (`_ambReconfigSharedQuiet` + old-only gate-ramp) is no longer used by the advance. DON'T re-gate the depart on `barLock`/`_barLockLoopBars` â€” Free layers have `_barLockLoopBars=0` and would fall through and never fade. Reason for departing to a temp key: a layer has ONE chain per key, so to overlap old-fading + new-fresh for the same key you must move the old one aside first. The advance times off `_barGridAnchor` (the lead grid, reset per advance), not `_orchStartAt`, AND fires EARLY with a GENEROUS margin (~0.6 s lead) so the advance always fires BEFORE the boundary even under tick jank (the tick handler can take 50â€“70 ms). It passes the precise boundary to `_ambOrchAdvance(E, boundaryAt)`, which sets `at = boundaryAt > now ? boundaryAt : now` and cancels pending voices + starts each layer's fade AT `at`. CRITICAL: the lead must be big enough that `at` is the FUTURE boundary, not a late `now` â€” if the advance fires AFTER the boundary, `at` falls back to `now` and the outgoing SYNCED layers (which can't cancel already-started voices) bleed past the boundary = overflow. A too-small lead (0.3 s) caused exactly that under jank. The incoming area is anchored to the boundary (`E._barGridAnchor = at`), so its synced layers' first onsets land on the next bar AT the boundary, not before it (overlap) â€” both the outgoing cut and the incoming start sit on the boundary.

- **Adding a mod LFO "Shape" needs THREE places in sync, or it silently degrades.** `_ambSyncTarget` validates the shape against the `_AMB_MOD_SHAPES` whitelist and falls back to `'sine'` if absent â€” so a new shape added only to `_ambShapeSel` (the UI) plays as sine. Update all three: (1) `_ambShapeSel` (UI option), (2) `_AMB_MOD_SHAPES` (whitelist), (3) `_ambMakeSrc` (how it's built). Native Tone.LFO types (`sine/triangle/sawtooth/square`) just pass through as `type`; non-native shapes need handling â€” `rampdown` is a `sawtooth` with min/max swapped (inverts direction); `smooth`/`sharp` are random/sample-&-hold via a stepped `Tone.Signal` (`_ambIsStochastic`); `custom` is a harmonic/partials wave (4 amps in `mod[t].partials`) sampled to a sum-of-sines curve and ramped through a `Tone.Signal` by `_ambScheduleCustomSrc` (reusing the seq-curve stepping path, `kind:'custom'` in `_ambSyncTarget`/`_ambMakeSrc`). The custom partials row + bindings live in the SHARED `_ambModTarget`/`_ambWireModTarget`/`_ambSyncModShapeEl`, so one edit covers all layer types; default partials `[100,0,0,0]` == a sine (so it's harness-neutral and a sane default).
- **Sample/pad voices register as ACTIVE immediately (even when scheduled ahead) â€” so future-scheduled ones live in the ACTIVE sets, not the pending sets.** This bit the area-transition departure: `cancelBloomFutureVoices(key, fromAt)` only scanned `_pendingVoices`/`_pendingSampleVoices`, so a sample voice scheduled to start AT/AFTER the boundary (in `_activeSampleVoices`, `_akAt >= fromAt`) was neither cancelled (not pending) nor stopped by `stopBloomVoicesBefore(key, beforeAt)` (which only stops `_akAt < beforeAt`). It then started in the NEXT area and piled up across transitions = "samples run over and get louder". Fix: `cancelBloomFutureVoices` now ALSO scans `_activeVoices`/`_activeSampleVoices`/`_activePadVoices` for `_akAt >= fromAt` (not-yet-sounded â†’ safe to cancel, no click). Synth voices mostly defer to `_pendingVoices` so they were already caught; SAMPLES are the ones that leak. Diagnose voice leaks with the `_activeSampleVoices.size` / `_activePadVoices.size` totals.

- **A LOCKED layer's notes render as an EDITABLE piano-roll canvas (`lockroll` mode), not chips.** `_ambUpdateNotesLive` switches a `.ambient-notes-live` line to a `<canvas class="ambient-np-roll ambient-np-lockroll">` when `_ambLockedNotes` returns notes (which it does for ANY frozen layer, incl. Bar-Lock `_barLock` â€” so Bar Lock auto-exposes editing = item "#3B" needs no extra code). `_ambDrawLockedRoll` draws the loop STATIC over its period (`_ambLockMeta` gives list/offKey/durKey/P) with a playhead from `E.freeze[key].anchor`; `_ambWireLockedRoll` hit-tests a pointerdown to a note index â†’ `_ambOpenNoteEditor` (click block = edit) or `_ambAddNoteAt` (click empty = add there). GOTCHA: the editor-dismiss check in `_ambUpdateNotesLive` must NOT require a `.ambient-np-note` chip (there are none in roll mode) â€” it falls back to validating `_ambNoteEd.index` against `_ambLockedNotes(...).list`, or the popover closes itself one frame after opening. Drawn every frame by `_ambDrawRolls` (canvas created once on mode-change, redrawn in the viz rAF). Pure UI/edit (edits apply via `_ambReemitLockedNext`), so the note-generation harness is unaffected.

- **The "Riff" layer is internally the `run` type (display rename only).** `_AMB_LAYER_SCHEMA.run.label` and the Add-layer menu say "Riff", but the config key, `_ambEmitRun`, `E.runPhase`, the period/notesOf maps, and saved projects all still use `'run'`. Don't rename the TYPE â€” it would orphan saved projects and break the harness. Riff's controls map: # notes = DensityĂ—Bars, # bars = Bars, rests = Rests, note-length distribution = **Len var** (`lenVary`, via the gated `_ambVaryLen` â†’ no rng draw at 0, so the default stays harness-identical).

- **The Bloom tick captures `cfg` ONCE at the top; the orch-advance runs MID-tick and swaps the play area â€” so after firing the advance the tick MUST `return`, or it re-generates the OUTGOING area with the stale `cfg`.** `_ambTick` reads `cfg = _ambPlayCfg(E)` (the play area) near the top, then the orch boundary check (`now >= boundary - lead`) calls `_ambOrchAdvance` which swaps `E._playIdx` to the new area, cancels the outgoing layers' voices, AND generates the new area (its own `_ambTick`). If the original tick then CONTINUES into its layer-generation section, it emits the OLD area's layers again using the captured (stale) `cfg` â€” those voices are scheduled INTO the new area's time and AFTER the cancel ran, so nothing cancels them â†’ they bleed into the next area (symptom: "bass/pedal play in the next area even though it has no bass/pedal layer"). Fix: `return` immediately after `_ambOrchAdvance` (the advance already generated the new area; `_orchSwitching` is reset at the end of the advance so the next tick proceeds normally). This is why the depart/cancel/stop machinery all looked correct in isolation but layers still overflowed â€” the re-emission happened downstream of all of it.

## Layer-schema evolution rules (Bloom)

Bloom is moving toward a **composable-layers** model (a layer = Voice Ă— Note-source Ă— Generator Ă— Timing Ă— Variation Ă— Mix/FX; today's ~11 types become presets). Full design + phased plan: `docs/bloom-composable-layers.md`. Until those phases land, the invariants below govern any change to a Bloom layer's saved shape:

1. **`_normalizeAmbientCfg` is the ONE migration chokepoint.** Every load path (Drive, localStorage, `_cloneLane`, undo snapshots, Send-to-Bloom, Shape-It, area switch, play) funnels through it. Put migrations there, nowhere else.
2. **Each area cfg carries `schemaVersion`** (`_AMB_SCHEMA_VERSION` in `17-ambient.js`; legacy/pre-versioned = 0). A migration reads `_fromVer` at the top of normalize and gates on `_fromVer < N`; the stamp before `return cfg` records the current version so the migration runs **once** and re-normalizing is a no-op. Bump the constant + add a `//   vN â€” â€¦` comment line ONLY when a layer's meaning/shape changes and needs a one-time migration.
3. **Additive-only; never repurpose a field.** Add new fields with defaults; don't change what an existing field means. An old build reading a new project must degrade safely (unknown fields normalized away).
4. **Backwards compatibility is an invariant, not a migration project.** A legacy layer with only `type` must keep producing byte-identical output. When Phase 2 adds `generator`/`voice`, they are *derived* from `type` on load and dispatch falls back to `type` â€” same core, same params.
5. **The invariant harness (`23-bloom-harness.js`) is the gate.** It hashes `[at,freq,durMs,voice,pan,level]` per note over seeded RNG/fixed ticks. Any schema change must keep it green; an intentional output change is a deliberate, reviewed baseline bump â€” never silent drift. Adding/removing an `_ambRand()` call on a covered path shifts every downstream draw, so keep RNG draws identical on default paths.

## Architecture

This is a browser-based music player that streams audio files from Google Drive. It is a single-page app served by a minimal Express static server (`server.js`).

### Module structure

Four vanilla JS modules are loaded in order via `<script>` tags in `index.html`:

1. **`js/google-drive-api.js`** â€” `GoogleDriveAPI` class. Initializes `gapi` and Google Identity Services (GIS) OAuth client, handles sign-in/sign-out, searches Drive folders, paginates through audio files, and fetches artist metadata.
2. **`js/music-player.js`** â€” `MusicPlayer` class. Wraps the HTML5 `<audio>` element, manages blob URL lifecycle (creates on play, revokes on next load), and handles keyboard shortcuts (Space, Arrow keys).
3. **`js/playlist-manager.js`** â€” `PlaylistManager` class. Owns the track array, renders the playlist DOM, handles drag-and-drop and touch reordering, and implements shuffle (Fisher-Yates, preserves original order for restore).
4. **`js/app.js`** â€” Wires everything together, manages auth UI state, loads playlist options from a Drive file at `bloops/playlists`, and shows loading/toast feedback.

### Event-driven communication

Modules communicate via custom DOM events (no external event bus):

| Event | Source | Consumer |
|---|---|---|
| `authStatusChanged` | `GoogleDriveAPI` | `app.js` (updates UI) |
| `trackLoaded` | `MusicPlayer` | `app.js` |
| `trackEnded` | `MusicPlayer` | `PlaylistManager` (advance track) |
| `requestNextTrack` | `MusicPlayer` | `PlaylistManager` |
| `requestPreviousTrack` | `MusicPlayer` | `PlaylistManager` |

### Key data flows

**Auth:** User clicks connect â†’ GIS OAuth popup â†’ access token stored in `GoogleDriveAPI` instance â†’ `authStatusChanged` fires â†’ UI updates.

**Loading music:** Folder name selected/searched â†’ `driveAPI.searchFolders()` â†’ `driveAPI.getMusicFilesFromFolders()` (paginated) â†’ optionally `driveAPI.fetchArtistName()` from a special "Artist name" file in the folder â†’ `playlist.setTracks()` renders list â†’ first track auto-loads.

**Playback:** `PlaylistManager.playTrack(index)` â†’ `MusicPlayer.loadTrack()` â†’ fetch audio blob from Drive with `Authorization: Bearer <token>` header â†’ blob converted to object URL â†’ set as `<audio>` src â†’ play.

### Google Drive folder convention

The app expects:
- A `bloops/` folder at the root of My Drive with subfolders `bloops/effects`, `bloops/projects`, `bloops/exports`. The Bloops sign-in init creates these on first run with one consolidated confirmation prompt.
- A text file at `bloops/playlists` listing folder paths (one per line) that the Player turns into playlists. Paths use `/` for parent/child nesting (e.g. `bloops/exports`). The init step seeds this file with `bloops/exports`.
- Audio folders optionally containing a file named **"Artist name"** whose first line is used as the artist, and an image named **"Artwork"** used as album art.
- Supported audio formats: `mp3, wav, flac, aac, ogg, m4a, opus`.

### Google API credentials

`CLIENT_ID` and `API_KEY` are hardcoded in `js/google-drive-api.js`. To use a different Google Cloud project, replace those constants and update the authorized JavaScript origins in the OAuth client configuration. The app uses the `drive.readonly` scope.

### Deployment

`deploy.sh` stages the project, installs production dependencies, and uploads via SFTP using `lftp`. See `pre_deploy.txt` for GoDaddy cPanel prerequisites (Node.js app setup, SFTP credentials).
