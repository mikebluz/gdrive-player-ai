# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## UI guidance

Apply these rules to every UI change:

1. **No horizontal scrolling.** If a row's contents don't fit, either (a) shrink elements in that row to fit (`min-width: 0`, `flex-shrink: 1`, narrower padding/font on small viewports via media queries), or (b) wrap to a new row and resize the new row's contents to fit. Never solve overflow with `overflow-x: auto/scroll`.
   - **Lane step strips (`.lane-chips`) are a BAR GRID** (this replaced the old single-row scrolling timeline — the prior carve-out is gone). Each `.lane-chips` is a CSS grid `repeat(_BAR_SUBCELLS, 1fr)` where **1 row = 1 bar** = `_BAR_SUBCELLS` (32) sub-cells, fitting the strip width with **no horizontal scroll**. A step spans `round(stepLengthFactor × 32)` cells (one whole note = a full bar/row); a step crossing a bar line **splits into continuation segments** (`.cont-start` head + `.cont-mid`/`.cont-end` clones, marked `.cont-segment`) via `_barGridPlan`. Continuation segments are visual-only (no handlers, excluded from index-based selectors by `:not(.cont-segment)`). The old horizontal sweeping playhead cursor (`_followChipInLane`/`_positionCursorsAtTick`) is disabled — playback shows via the per-chip `.active-loop` highlight. Triplets/odd meters: 32 isn't divisible by 3, so triplet steps round to the nearest cell — they're now MARKED (a ³ badge + amber accent + a bracket tick per group of 3, via `_isTripletStep`/`.triplet`) rather than made exact (an exact grid would need 96 cells). The grid playhead is DONE: a line sweeps each `.lane-chips` strip per bar row, driven by a rAF loop off the audio clock (`_startBarPlayhead`/`_barPlayheadFrame` in 07, positioned from `_intrinsicLoopSec` + the fixed grid geometry).

2. **Audio playback must be smooth and immediate.** Every user-triggered sound (cell press, wrap audition, sequence/loop playback, REST tap, sample preview, recording playback, etc.) should fire with no perceptible lag, no distortion, and no fluctuation. When changing audio code, prefer firing the sound *before* DOM work, keep Tone's `lookAhead` minimal (currently 25 ms in `bloops.html`), and don't add unconditional time cushions to interactive triggers — guards for cold-start (suspended `AudioContext`) are fine but should not penalize the warm path. If a change risks audio glitches (under-runs, stagger between voices, dropouts), test it in the browser before declaring done.

3. **Keep the audio signal-flow diagram current.** `README.md` (section *"Bloops — audio signal flow"*) holds an ASCII diagram of how notes route from each source through the two entry buses (`globalSendTap` vs. the per-lane bus) to the shared master chain and speakers. Whenever you change audio routing — `playNote()` destination resolution, `globalSendTap`, `getLaneBus()`, the FX send/return wiring, or the master chain order — **update that diagram in the same change** so it never drifts from the code. Prefer one global routing rule over per-mode volume/FX hacks; if modes sound inconsistent, fix it at the bus/master level (and re-draw the diagram) rather than scaling individual call sites.

## Record learnings here (avoid repeat debug churn)

When a bug or behavior takes more than a couple of attempts to figure out — especially anything non-obvious about this codebase (CSS that overrides JS, event/lifecycle ordering, audio scheduling/lookahead, save/load quirks, the Bloom engine, view-mode rules) — **add a short note to the "Gotchas / hard-won learnings" list below in the same change**, so the next task doesn't rediscover it. Keep each entry one or two lines: the symptom and the fix/cause. Check this list before deep-debugging a UI/audio/state issue.

### Gotchas / hard-won learnings
- **The euclid Pattern grid can load rhythm PRESETS (`_EUC_PRESETS` from 09, the same list the grid "Generate Sequence" dialog offers) — euclidean presets drive the knobs, clave presets store an override.** `_ambEuclidPresetOptions`/`_ambApplyEuclidPreset` (17) group `_EUC_PRESETS` by `grp` (1 World / 2 Rock / 3 African / 4 Clave) into a `<select>` in the Pattern control. A preset with `{k,n,rot}` sets Steps=n, Pulses=k, Rotate=rot and CLEARS the override (the generated euclidean pattern IS the preset, so multi-voice spread still works); a preset with a `pat` string (non-euclidean clave/bell timeline) sets Steps and stores `euclidPattern=[row]` on voice 0. `_EUC_PRESETS` is a top-level `const` in 09 (loaded before 17) so it's visible cross-file — guarded with `typeof` anyway. To fit presets up to n=24 the euclid **Steps max was raised 16→32 and Rotate 15→31** (schema rows + ALL emitter clamps `Math.min(16→32, …steps)` in `_ambEmitEuclidCore`/`_ambEmitArpEuclid`/the readout/`_ambEuclidGridDims`) — harness-safe (defaults are ≤8, so `Math.min(32,8)===Math.min(16,8)`). The grid also shows a **Bars readout**: `_ambEuclidBarInfo` — bar count = **Phrase** (`bars`); when Unit-Synced it appends "· plays in N bars" (Lock-to time-scales the phrase's DURATION, a separate axis from its bar COUNT). If you add a preset kind or change the step cap, update `_ambApplyEuclidPreset` AND keep the four `Math.min(32,…)` step clamps in lock-step.
- **Euclid layers (bass, euclid Beat, euclid Arp) carry an editable step grid spanning the WHOLE PHRASE — `inst.euclidPattern` (per-voice rows) OVERRIDES the generated euclidean pattern, with PER-BAR editing.** All euclid emitters resolve their per-slot pattern through `_ambEuclidPat(inst, pulses, steps, rotate, V, v, salt)` (NOT `euclideanPattern`/`_ambEuclidVoicePat` directly) — it returns `inst.euclidPattern[v]` when that row's length is a nonzero **multiple of `steps`** (one bar-block per Phrase bar, or a single-bar override), else the generated (steps-long) pattern. **Emitters index `pat[(bar*steps+slot) % pat.length]`** so a steps-long pattern REPEATS every bar (the default → byte-identical to before, harness green, ZERO added rng draws) and a phrase-long override plays a different block per bar. The grid (`_ambEuclidCellsHtml`/`_ambWireEuclidGrid`, schema token `euclidgrid`, shown in euclid mode via `_ambBeatGenVis`/`_ambArpEuclidVis`) renders V voice-rows of **`bars`×`steps`** cells (bar groups w/ `.barstart` dividers + `data-bar` badges); each cell's `data-ei` is its ABSOLUTE index `bar*steps+slot`. A cell tap SNAPSHOTS the whole phrase (generated repeats via the same modulo) into a `bars*steps`-length override then toggles. **Steps** change + Regen CLEAR the override (`_ambEuclidRegen` nulls it); Pulses/Rotate also clear; **Phrase (`bars`) and Voices KEEP** it (grid re-renders, editing re-snapshots to the new length). The bars readout re-renders on Timing edits via `_ambRefreshEuclidGrids` (hooked into `_ambSyncLayerUnits`, which Lock-to calls) + direct `bars`/`rate` listeners. Normalized in the one chokepoint (`_ambNormalizeEuclidPattern`) — keeps any non-empty 0/1 rows (length checked at read time). Raising Phrase LENGTHENS the grid. If you add a 4th euclid call site, route it through `_ambEuclidPat` AND use the `(bar*steps+slot) % pat.length` indexing, and keep `_ambEuclidGridDims` (now returns `bars` too) in lock-step with the emitter.
- **The core-strip reverb-send bus (worklet output 16) is GLOBAL — one summed send, feeding ONE reverb — and must be (re)claimed by the engine that's generating.** Symptom: "turn a layer's Reverb send up, hear no change" on lane/Shape Bloom under core strips: `_ambEnsureReverb` used to connect the bus to the MASTER engine's reverb only (often never built), and only at reverb creation — a pre-worklet claim failed silently and never retried because the reverb caches. Now `connectSend` (03b) routes through a persistent native gain (output 16 itself is never disconnected — the keep-pull sink hangs off it) and `_ambBuildMod`'s core path re-claims the bus on every strip build, so the last engine to build strips owns it. Diagnose send-path deadness with analysers on worklet output 16 (send sum) vs the engine's reverb output: send hot + reverb silent = orphaned bus, both silent = the strip isn't sending.
- **Key-grouped chips MUST render inside a subgrid container — a flex `.key-group-container` silently shrinks them to text width.** Steps sharing a `keyContext` get re-parented into `<div.key-group-container>`; inside a flex box their `grid-column: span N` is ignored, so grouped quarter-note chords drew ~2 cells while playing (and cursoring) a true 1/4 — an audible-vs-visual divergence that ONLY hits key-grouped steps (injected test steps without keyContext looked fine, which hid it). The old `.grid-mode` gate on the subgrid rule died when the bar grid landed, dropping every container into the flex fallback. The rule is now unconditional; the container's column span is clamped to the row (over-span forces implicit columns and shrinks the whole strip) with `grid-row: span ceil(total/32)` for multi-bar groups. Diagnose suspected width bugs by comparing a step WITH vs WITHOUT `keyContext`.
- **NEVER re-enable synth body-pooling (`VOICE_POOL_ENABLED`) without passing the oscillator-leak test — pooled retriggers leak RUNNING native OscillatorNodes (1-2/note), and THAT was the dense-project cut-out.** Tone's retrigger bookkeeping cannot reliably stop the previous note's native oscillator when a synth body is reused (native source stops can't be cancelled; restarts orphan running nodes; stopping at pool-release makes it worse). A started-never-stopped OscillatorNode is GC-protected and renders forever: ~25 leaked/s on an all-FM project drowned the render thread in ~12 s (audio clock < 1× wall clock = the device plays gaps = glitch → silence; Stop→Play recovers because voices get disposed). Diagnose node leaks with puppeteer `page.queryObjects(OscillatorNode.prototype)` after `window.gc()` — flat counts good, linear growth = leak. Fresh-synth-per-note + `safeDisposeSynth` leaks zero; construction cost is handled by the deferred voice-build queue, NOT pooling.
- **The dense-project cut-out was RENDER CAPACITY, machine-dependent — the voice budget now ADAPTS to the measured audio clock; don't re-diagnose silence as engine bugs before reading `[bloom-health]`.** On the user's machine the audio clock ran at 0.86× wall clock (device playing gaps = glitch→silence, meters still flashing) with only ~19 plain FM voices — below the static 24-unit cap — and voice tails hit steady-state concurrency ~12 s after Play (bed ≈ 12.3 s lifetime), so it always died "around 12 seconds". The Bloom health watchdog (`_ambHealthStart`, 17) measures ctRate every second with signal probes bracketing the master chain, and feeds `_setVoiceBudget` (04): ctRate <0.95 → shed most-decayed tails to 70% of current cost (floor 8); >0.985×3 s → recover +1 (session-persistent). Console prints `[bloom-health] render behind … voice budget → N`. Diagnostic buffer: `window.__bloomHealthLog`. If a future silence report shows healthy ctRate AND healthy signal probes, THEN hunt routing/NaN — the probes name the dead stage.
- **NEVER connect a modulator signal into a biquad filter param, and never run continuous ramps on one — schedule stepped/glide control-rate curves instead.** Measured in Chrome (audio-clock rate at 24 voices): connected LFO→`filter.frequency` = 0.82× real time, continuous linearRamps = 0.96, stepped/5ms-glide points = 1.00 (free). A signal input (or an always-active ramp) forces per-SAMPLE biquad coefficient recompute; `automationRate='k-rate'` is silently ignored by Blink. So the Design mod-matrix applies cutoff/reso as ONE pre-summed scheduled curve (filter-env + all mods, sampled every 45 ms with 5 ms glides — `_sdBuildModRig`'s `scheduleCurve`, env math shared via `_sdFilterEnvShape`), and per-voice deterministic LFOs for SEQUENCED notes are scheduled `Tone.Signal` automation, not `Tone.LFO` nodes (2 LFO nodes × 24 voices alone = 0.57× real time). Pitch/amp/pan mod stays as connected signals (measured free). Held presses (dur>30 s) keep true LFOs/connected topology — one live voice is affordable. The VOICE CAP is a DSP **budget**, not a headcount: `_voiceDspCost` weights design voices (filter +1, matrix +0.5, sub/ring +0.5, unison +0.15/voice) so plain projects keep the old count exactly while design-heavy stacks settle at a render-thread-safe concurrency.
- **Scheduled-ahead voices BUILD via a deferred time-sliced queue inside playNote — new cancel/stop paths must purge it, and emit-tag globals are captured per entry.** playNote defers construction of any voice with ≥0.25 s lead (all Bloom notes; interactive presses build synchronously) into `_voiceBuildQueue`, drained ~10 ms per 16 ms pump, earliest `at` first (`_vqPump`). This is what actually stops dense-project cut-outs: even POOLED voices cost ~2-3 ms each to wire+trigger, so a 20-voice chord/burst built synchronously in one tick starved Tone's scheduler no matter how cheap each voice got. Notes still sound at their exact `at`; only construction moves. THREE invariants: (1) any new cancel/stop/fade path that sweeps `_pendingVoices` must also `_vqPurge`/`_vqClear` the queue or cancelled notes still build and sound; (2) voice `_ak` tagging is temporal (reads `window._ambEmitKey`/`_ambEmitAt`/`_ambCaptureSink` at build) so entries capture those globals at enqueue and `_vqBuild` restores them around the build — any new emit-time side-channel global read inside `_playNoteNow` needs the same treatment; (3) the capture-sink TEE runs at emit time (before the defer gate) — keep it there or Freeze/Shape-It capture breaks. Urgent entries (<0.3 s to onset) get a BOUNDED 30 ms slice — never unlimited, so a backlog degrades to slightly-late notes instead of a scheduler-starving burst. Kill switch: `VOICE_BUILD_QUEUE_ENABLED`.
- **Design patches must POOL their synth bodies, and pool release keys must match acquisition keys.** Two dense-project cut-out causes fixed together: (1) `_sdOscNeedsFreshVoice` used to force a fresh voice for any fm/am patch with a finite harmonicity — but EVERY Design patch carries one (default 3), so Design FM/AM built a full synth (~7ms) per note and a bed+motif+riff stack starved the scheduler → silence. FM/AM timbre is now set per acquisition on the pooled body, and unison designs pool under a signature key (`basic:sine:u5:s30` — FatOscillator rebuilds when `count` CHANGES, so configs can't share a bucket); only ring-mod still needs a fresh voice. (2) playNote released pooled synths under the raw `type` while basics acquire under `'basic:'+type` → released synths parked in a dead bucket and basics never actually reused. Release sites now use the `_poolKey` stamped at acquisition — never hand-derive a pool key.
- **The EUCLID arp anchors phase in `E.runPhase[key]`; the SERIES arp in `E.arpState[key]` — don't conflate them.** Both are `type==='arp'`, but the euclid arp (`arp.euclid`) is windowed/phase-anchored like Bass/Beat-euclid (`_ambEmitArpEuclid` reads/writes `E.runPhase[key].lastAt`), while the series arp uses `E.arpState[key]`. This bit the Bar-Lock UNLOCK: `_ambResumePhaseAt` (thaw handler) reset only `arpState` for arps, so a thawed euclid arp's `runPhase.lastAt` stayed stale → live gen resumed at `now` and doubled the still-playing captured loop from unlock until the next boundary. Fix: discriminate by `L.euclid` and re-anchor `runPhase[key].lastAt = tA`, CREATING `{startAt:tA,lastAt:tA}` when the locked run left it absent (else the emit rebuilds it with `lastAt=null` → `tFrom=now` → the `hh.at<tFrom` per-hit guard no longer holds the line at the boundary). Any new per-arp phase/state code must branch on `L.euclid` for the store.
- **Bar Lock loop length = the manual `cfg.bars` (a hard override), NOT the LCM.** Early Bar Lock auto-sized the loop to `_ambAreaLoopBars` (LCM of capturable layers) and HID the Bars picker — so Bars did nothing while locked. Now `_ambBarLockSync` captures over `cfg.bars` and `_ambAreaDurSec` = bars×plays (locked or not). `_ambAreaLoopBars` survives only as (a) the "nothing capturable → stay live" guard and (b) the advisory **`◈ N-bar unit`** readout (`.ambient-orch-unit`) beside Bars — grey when `bars % unit === 0`, amber `.warn ⚠` otherwise (a non-multiple cuts a layer mid-cycle at the loop seam = the user's informed choice). If you touch the loop-length math, keep the LCM as advisory only; don't re-hide Bars.
- **Wrap-ensemble = a wrap with PER-NOTE TONES, voiced by Bloom; the LAYER MODE supplies the shape.** Wraps always gave Bloom only PITCH (`intervals`+`root`); the wrap-ensemble adds per-degree TONE. `_ambPublishWrap` captures `wrap.tones[]` (parallel to `intervals`) from the wrap's chord notes that carry `toneOverride`+`sound` (set in the Seed wrap step-editor — already existed). `_ambDegreeToneAt(deg, src)` returns a wrap's per-degree tone (null otherwise). Two delivery mechanisms: (1) SINGLE-note emits read a side-channel — `_ambDegreeFreq` stamps `_ambLastDegTone` on every pick, and bass/run/motif/pedal/texture do `if (_ambLastDegTone) params.type = _ambLastDegTone` right after the pick (works only because the pick immediately precedes the param build, no intervening pick). (2) MULTI-pick emits read INLINE (the side-channel would be stale): bed `_ambPickVoicing` chaos path builds `{f,t}` pairs, sorts by freq, and hangs a parallel `freqs._tones` on the returned array which `_ambEmitBed` applies per voice (= the **Stack** shape; structured Chords/Monk modes use chord FORMS not wrap degrees, so they don't carry tones); arp series uses `_ambDegreeToneAt(d, notes)` with the interval index `d`; arp euclid stores `tone` on each precomputed hit at build (its `deg` isn't in scope at play). **The shape is the layer**: single-note layers → Run (a melodic line, each note its own timbre), bed chord → Stack. **Gated/harness-safe:** every site applies the tone ONLY when non-null, so a non-wrap source (or a wrap with no per-note tones) is byte-identical to before — the note stream (and the invariant harness, which doesn't hash tone) is unchanged. NOT covered: the bed STRUCTURED modes (form-based, not degree-based).
- **Body-attached overlays get force-hidden by view-mode CSS.** Rules like `body.<mode> > *:not(#mix-view):not(.ctx-menu) { display:none !important }` (and the `tracks-fullscreen` family) hide any element appended to `<body>` that isn't exempt. A selector `!important` beats a normal `.open{display:block}` rule, so toggling a class won't show it. Fix: set `display` via **inline `!important`** (`el.style.setProperty('display','block','important')`); clear on close. Diagnose with a toast of `getComputedStyle(el).display` + `document.body.className`.
- **`position:fixed` overlays drift** because the app uses `backdrop-filter` on some ancestors, which makes one a *containing block* — so `getBoundingClientRect`-derived `top/left` are offset (and shift with scroll). Fix: **measure-and-correct** — set the overlay to `0,0`, read its rect (= the containing-block offset), then subtract it from the desired viewport coords.
- **Live-readout chips must not be `<button>` / focusable.** The notes readout repaints on a timer, so a `click` (which needs mousedown+mouseup on the *same* element) gets dropped when the element is recreated between press and release; and focusing a control inside an `aria-hidden` line is blocked. Fix: render chips as non-focusable `<span role="button">` and act on **`pointerdown`**, not click.
- **Bloom edits land one iteration late** unless you account for the ~1.4 s lookahead: the next iteration's voices are already scheduled with old values. To apply on the next iteration, cancel the layer's scheduled-ahead voices and re-anchor/re-emit. Use `cancelBloomFutureVoices(key, fromAt)` (voices are tagged with `_ak`/`_akAt`) to drop *only* the next iteration so the current one finishes intact.
- **Silent/dry Bloom render** already exists: set `window._ambSilentCapture = true` (with a capture sink) and `playNote` captures without sounding — reuse it instead of adding a dry-run path.
- **Always-on per-layer DSP glitches modest Bloom stacks.** Continuous DSP built for every layer even when inactive (it ran at flat/0 settings) drains the audio render thread and glitches/cuts out with only a handful of layers. This is why Distortion/Delay are spliced in LAZILY by `_ambApplyLayerFx` only when `mix > 0`, and why the per-layer **EQ3 is lazy too** (`_ambApplyEq` inserts an EQ3 between vcf→vca only when a band ≠ 0; the meter FFT analyser stays on the VCA so meters still work). Rule: don't add per-layer nodes that run unconditionally — gate them on the feature being engaged, or a 5-layer project chokes. Even *engaged* heavy DSP needs scaling: per-layer **Distortion oversampling** is adaptive (`_ambDistOversample`/`_ambNormalizeDistOversample`) — 1 dist layer = 4x, 2-3 = 2x, 4+ = none — because 5× 4x-oversampled waveshapers (the "FX on all layers" case) overran the render thread and cut audio out (recovered only by Stop→Play, which tears the FX down).
- **A layer's `level` has two synced controls and ~5 writer paths.** The mixer fader and each layer card's "Level" slider are both shortcuts for the same `layer.level`. Writers: mixer fader (`_ambRenderMixer`), primary `bind()`, extras schema `k==='sl'`, seq `bindInt`, sample `bindInt`. Each calls `_ambSyncLevelUI(E, key, v)` to mirror the value into the OTHER control's slider + readout (it only sets `.value`/text — no synthetic `input` dispatch, so no feedback loop). The mixer fader carries `data-mixkey=<layer key>`; cards are found via their header `[data-phkey]`. If you add another place that mutates `level`, call `_ambSyncLevelUI` there too or the two controls drift.
- **Bloom Level is a CONTINUOUS per-layer GAIN node (`e.levelGain`), NOT a per-note volume.** Chain: `vcf → vca → levelGain → gate → pan` (`levelGain` spliced in `_ambBuildMod`). `_ambApplyLevel` is now a NO-OP pass-through; the level is applied as a gain via `_ambLevelGain(level)` (0→silent, 70/default→×1.3 baseline lift, 100→×4 / +12 dB — `_AMB_LEVEL_BASE`/`_AMB_LEVEL_MAX`), written by `_ambUpdateMod` (build/sync), `_ambSyncLevelUI` (every manual writer — real-time fader), and the ramp's `key==='level'` `set` (real-time sweep). The ceiling is ×4 (not the original ×2) BECAUSE several layer types are per-note-staged low (`_ambBassParams` ×0.34, run/pedal ×0.32, drone ×0.2 in their param builders) — the old ×2 couldn't lift them to parity, so quiet layers stayed quiet even maxed; ×4 gives the slider enough range (0.34×4=1.36 > 1). Those base stagings are HARNESS-HASHED (`params.volume`), so boost via `_ambLevelGain` (the gain node) — NOT the base volumes — to stay harness-safe. **Why:** per-note level only affected NEW notes, so on a sustained/overlapping pad the held tails masked it ("the level ramp does nothing"); a layer gain sweeps the WHOLE layer, tails included, in real time. The reverb send now taps `levelGain` (still pre-gate → scales with level AND tails ring through a gate mute). **Harness-safe:** at the default level 70 the no-op equals the old curve exactly, and no harness config sets a non-default level, so the invariant battery is byte-identical. If you add a new `level` writer, push to `e.levelGain.gain` (via `_ambSyncLevelUI`) or held voices won't follow.
- **The TRANCE GATE is a bar-synced step pattern on a dedicated chain node (`e.tgGate`), driven by a `Tone.Signal` (`e.tgSig`).** Chain: `vca → levelGain → tgGate → gate` (tgGate spliced in `_ambBuildMod`, pre-gate so the reverb — tapped off `levelGain` — fills the chopped gaps). Config is per-layer in the FX object: `tg:{on,steps,pattern[],depth,edge}` (`_ambDefaultFx`/`_ambNormalizeFx`; `steps` = steps/bar = pattern length). The driver `_ambScheduleTg` (called per layer from `_ambScheduleStochastic` each tick) schedules the pattern AHEAD onto `tgSig`: **`tgGate.gain` intrinsic is 0 and the connected `tgSig` sets it** (a connected signal SUMS with the param, so intrinsic must be 0, not 1). When `tg.on` is false it disposes `tgSig` and parks `tgGate.gain=1` (passthrough). The grid is anchored to `E._playStartAt` at the cfg `bpm` (`barSec=(60/bpm)*4`) so it locks to the bar. `tgSig` is disposed on stop/teardown so the automation timeline can't accumulate across plays (the codebase's rule — a dedicated disposable Signal, never per-step automation on a persistent shared param). UI: a "Trance Gate" subsection (On/Steps/Depth/Edge + a step grid reusing `.ambient-slice-grid`/`.ambient-slice-cell`) that lives in **`_ambModUi`, grouped under the VCA mod** (both modulate gain — VCA = smooth LFO, gate = step pattern); its element ids are still `fx-tg-*` (unchanged when it moved out of `_ambFxUi`), so the shared `_ambWireTg(E, el, getL, persist)` — called from all four FX wire clusters (seq/sample/extras/primary) — still binds it by id. Harness-safe: `tg.on` defaults 0 and the gate only touches the gain node, never note generation.
- **Portamento (per-layer pitch glide) is GATED and only covers the OSCILLATOR voice path.** A layer's `portamento` (ms, shared default 0 via `_ambNormalizeSpread`) is read by `_ambApplyAdsr` (the shared params builder for all 13 Bloom emits), which — only when `portamento > 0` — sets `p.glideMs` + `p.glideLayer = inst`. In `playNote` (04, the wavetable/oscillator branch), when `glideMs > 0` each `Tone.Oscillator.frequency` is ramped from the layer's previous note freq (tracked on `glideLayer._glidePrev`) to the target over `glideMs`. **Gated** so the default path (and the invariant harness, which doesn't hash glide anyway) is byte-identical. **COVERAGE (all wired):** the glide is applied in THREE `playNote` voice paths — (a) oscillator/wavetable (ramp each `Tone.Oscillator.frequency`), (b) the `.frequency` synths FM/AM/Duo/Mono/Membrane/Metal (ramp `synth.frequency` after `triggerAttackRelease`), and (c) **samples** (ramp `source.playbackRate`, tracked on `glideLayer._glidePrevRate` — note the slice window is computed at the target rate, so a brief glide can drift the slice slightly). Pluck/Noise don't glide (no sustained pitch). The `portamento` property is on EVERY Bloom layer (`_ambNormalizeFx`) with a UI slider: primary layers (literal `sl`), seq+sample (`_ambSl`), extras (`_AMB_LAYER_SCHEMA` `['sl','portamento',…]`). **Seed (grid lanes)** are wired too: `lane.portamento` (`_makeLane`), injected into the scheduler's SINGLE-note step path (`07-playback-scheduler.js` — chords skip it, they'd share one `_glidePrev`), `glideLayer` = the lane object, control = a slider in `_showLaneMenu`.
- **Master Bloom decouples the VIEWED area from the PLAYING area.** `getCfg()` (= `masterAmbient` = `areas[activeIdx]`) is the area the PANEL shows/edits; the ENGINE generates `areas[E._playIdx]` via **`_ambPlayCfg(E)`** (returns getCfg for lane/shape/harness — `E._playIdx` is only set on the master while playing, so the invariant harness is unaffected). The two engine cfg sources are `_ambTick` and `_ambSyncMods` (both use `_ambPlayCfg`); generation reads the cached `E._cfg` which the tick sets from the play area. **So while playing you can click a tab to view/edit ANY area without changing playback** (`_ambSwitchToArea` is view-only when `_masterEng.timer`). Orchestration advances `E._playIdx` (NOT `activeIdx`) and does NOT rebuild the panel. Drawn every frame by `_ambDrawRolls` (canvas created once on mode-change, redrawn in the viz rAF). Pure UI/edit (edits apply via `_ambReemitLockedNext`), so the note-generation harness is unaffected.

- **The "Riff" layer is internally the `run` type (display rename only).** `_AMB_LAYER_SCHEMA.run.label` and the Add-layer menu say "Riff", but the config key, `_ambEmitRun`, `E.runPhase`, the period/notesOf maps, and saved projects all still use `'run'`. Don't rename the TYPE — it would orphan saved projects and break the harness. Riff's controls map: # notes = Density×Bars, # bars = Bars, rests = Rests, note-length distribution = **Len var** (`lenVary`, via the gated `_ambVaryLen` → no rng draw at 0, so the default stays harness-identical).

- **The Bloom tick captures `cfg` ONCE at the top; the orch-advance runs MID-tick and swaps the play area — so after firing the advance the tick MUST `return`, or it re-generates the OUTGOING area with the stale `cfg`.** `_ambTick` reads `cfg = _ambPlayCfg(E)` (the play area) near the top, then the orch boundary check (`now >= boundary - lead`) calls `_ambOrchAdvance` which swaps `E._playIdx` to the new area, cancels the outgoing layers' voices, AND generates the new area (its own `_ambTick`). If the original tick then CONTINUES into its layer-generation section, it emits the OLD area's layers again using the captured (stale) `cfg` — those voices are scheduled INTO the new area's time and AFTER the cancel ran, so nothing cancels them → they bleed into the next area (symptom: "bass/pedal play in the next area even though it has no bass/pedal layer"). Fix: `return` immediately after `_ambOrchAdvance` (the advance already generated the new area; `_orchSwitching` is reset at the end of the advance so the next tick proceeds normally). This is why the depart/cancel/stop machinery all looked correct in isolation but layers still overflowed — the re-emission happened downstream of all of it.

## Layer-schema evolution rules (Bloom)

Bloom is moving toward a **composable-layers** model (a layer = Voice × Note-source × Generator × Timing × Variation × Mix/FX; today's ~11 types become presets). Full design + phased plan: `docs/bloom-composable-layers.md`. Until those phases land, the invariants below govern any change to a Bloom layer's saved shape:

1. **`_normalizeAmbientCfg` is the ONE migration chokepoint.** Every load path (Drive, localStorage, `_cloneLane`, undo snapshots, Send-to-Bloom, Shape-It, area switch, play) funnels through it. Put migrations there, nowhere else.
2. **Each area cfg carries `schemaVersion`** (`_AMB_SCHEMA_VERSION` in `17-ambient.js`; legacy/pre-versioned = 0). A migration reads `_fromVer` at the top of normalize and gates on `_fromVer < N`; the stamp before `return cfg` records the current version so the migration runs **once** and re-normalizing is a no-op. Bump the constant + add a `//   vN — …` comment line ONLY when a layer's meaning/shape changes and needs a one-time migration.
3. **Additive-only; never repurpose a field.** Add new fields with defaults; don't change what an existing field means. An old build reading a new project must degrade safely (unknown fields normalized away).
4. **Backwards compatibility is an invariant, not a migration project.** A legacy layer with only `type` must keep producing byte-identical output. When Phase 2 adds `generator`/`voice`, they are *derived* from `type` on load and dispatch falls back to `type` — same core, same params.
5. **The invariant harness (`23-bloom-harness.js`) is the gate.** It hashes `[at,freq,durMs,voice,pan,level]` per note over seeded RNG/fixed ticks. Any schema change must keep it green; an intentional output change is a deliberate, reviewed baseline bump — never silent drift. Adding/removing an `_ambRand()` call on a covered path shifts every downstream draw, so keep RNG draws identical on default paths.

## The WASM audio engine (bloops-dsp)

Bloom voices, layer strips/FX, and sample playback render in a Rust→WASM core (`dsp/`, built by `dsp/build.sh` → `js/bloops/core/bloops-dsp.wasm`) inside ONE AudioWorklet (`js/bloops/core/voice-processor.js`, bridged by `js/bloops/03b-core-voices.js`). **Default ON** — `window.bloopsCore(false)` / `window.bloopsCoreStrips(false)` are the kill switches (persisted localStorage `'0'`); the Tone node engine remains the automatic fallback (cold start, ineligible notes, slot exhaustion, pads/held notes, offline export). Rules:
- **Any DSP change must keep `node test/golden-render.js` green** (72 bit-exact sections); an intentional audio change re-baselines with `--update` IN THE SAME COMMIT. `dsp/build.sh` runs the gate after every build.
- **Calibrate against RECORDED node output, never derive from Tone internals** (proven wrong repeatedly — see the engine memory file). Known engine truths: native lowpass/highpass biquad Q is in dB; `Tone.LFO/Signal.connect` ZEROES the destination param; cycle-member DelayNodes keep true delay with the quantum penalty on the feedback edge; `Tone.Panner` is channelCount 1 (mono-downmix — why the core strip's width-preserving pan law exists).
- The worklet must always stay pulled (keep-pull sink on output 16) and `init()` must reset ALL core globals (golden determinism).

## Architecture

This is a browser-based music player that streams audio files from Google Drive. It is a single-page app served by a minimal Express static server (`server.js`).

### Module structure

Four vanilla JS modules are loaded in order via `<script>` tags in `index.html`:

1. **`js/google-drive-api.js`** — `GoogleDriveAPI` class. Initializes `gapi` and Google Identity Services (GIS) OAuth client, handles sign-in/sign-out, searches Drive folders, paginates through audio files, and fetches artist metadata.
2. **`js/music-player.js`** — `MusicPlayer` class. Wraps the HTML5 `<audio>` element, manages blob URL lifecycle (creates on play, revokes on next load), and handles keyboard shortcuts (Space, Arrow keys).
3. **`js/playlist-manager.js`** — `PlaylistManager` class. Owns the track array, renders the playlist DOM, handles drag-and-drop and touch reordering, and implements shuffle (Fisher-Yates, preserves original order for restore).
4. **`js/app.js`** — Wires everything together, manages auth UI state, loads playlist options from a Drive file at `bloops/playlists`, and shows loading/toast feedback.

### Event-driven communication

Modules communicate via custom DOM events (no external event bus):

| Event | Source | Consumer |
|---|---|---|
| `authStatusChanged` | `GoogleDriveAPI` | `app.js` (updates UI) |
| `trackLoaded` | `MusicPlayer` | `app.js` |
| `trackEnded` | `MusicPlayer` | `PlaylistManager` (advance track) |
| `requestNextTrack` | `MusicPlayer` | `PlaylistManager` |
| `requestPreviousTrack` | `MusicPlayer` | `PlaylistManager` |

### Key data flows

**Auth:** User clicks connect → GIS OAuth popup → access token stored in `GoogleDriveAPI` instance → `authStatusChanged` fires → UI updates.

**Loading music:** Folder name selected/searched → `driveAPI.searchFolders()` → `driveAPI.getMusicFilesFromFolders()` (paginated) → optionally `driveAPI.fetchArtistName()` from a special "Artist name" file in the folder → `playlist.setTracks()` renders list → first track auto-loads.

**Playback:** `PlaylistManager.playTrack(index)` → `MusicPlayer.loadTrack()` → fetch audio blob from Drive with `Authorization: Bearer <token>` header → blob converted to object URL → set as `<audio>` src → play.

### Google Drive folder convention

The app expects:
- A `bloops/` folder at the root of My Drive with subfolders `bloops/effects`, `bloops/projects`, `bloops/exports`. The Bloops sign-in init creates these on first run with one consolidated confirmation prompt.
- A text file at `bloops/playlists` listing folder paths (one per line) that the Player turns into playlists. Paths use `/` for parent/child nesting (e.g. `bloops/exports`). The init step seeds this file with `bloops/exports`.
- Audio folders optionally containing a file named **"Artist name"** whose first line is used as the artist, and an image named **"Artwork"** used as album art.
- Supported audio formats: `mp3, wav, flac, aac, ogg, m4a, opus`.

### Google API credentials

`CLIENT_ID` and `API_KEY` are hardcoded in `js/google-drive-api.js`. To use a different Google Cloud project, replace those constants and update the authorized JavaScript origins in the OAuth client configuration. The app uses the `drive.readonly` scope.

### Deployment

`deploy.sh` stages the project, installs production dependencies, and uploads via SFTP using `lftp`. See `pre_deploy.txt` for GoDaddy cPanel prerequisites (Node.js app setup, SFTP credentials).
